# RestoPro ROS — Hybrid Cloud & LAN Architecture Manual
This guide details the deployment of RestoPro inside standard retail networks for zero-downtime operations, dynamic sync, conflict resolution, and hardware installation.

---

## 1. Network Topology (High-Availability LAN Setup)

Even when external internet circuits drop (fiber/broadband outage), standard physical branch terminals must continue registering orders, printing tickets, and managing active sessions. Output is synchronized to the Cloud as soon as WAN connectivity resumes.

```
                  [ CLOUD REGION (RestoPro Cloud Gateways) ]
                                    ▲
                                    │ (Continuous WAN State Synchronization)
                                    ▼
       ┌──────────────────────── LAN GATEWAY ROTUER ────────────────────────┐
       │                                                                    │
       ▼                                                                    ▼
┌──────────────┐                                                      ┌──────────────┐
│ BRANCH CORE  │ ◄──────────(WS/HTTP LAN Network Sync Channel)─────►  │ CAPTAIN APP  │
│ SERVER NODE  │                                                      │ Android / iOS│

└──────┬───────┘
       │
       │ (Raw TCP port 9100 Print Routing via Esc/Pos)
       ▼
 ┌──────────┐
 │ Thermal  │

 └──────────┘
```

The branch server executes a local node container alongside a local MongoDB instance.

## 2. Sync Fault Recovery & Conflict Resolution

During an internet disruption, local databases hold master authorities for new orders. On network resumption:
1. **Delta-Polling**: Devices call the Delta Sync endpoint (`/api/captain/sync-engine/delta`) passing their last known sync timestamp.
2. **Deterministic Sequence Merging**:
   - **Static Configuration (Menu, Taxes, Layouts)**: Cloud is always the master. Local records are overwritten.
   - **Dynamic Transactions (Orders, Tables, Stock Levels)**: 
     - If changes are non-overlapping (e.g. Table 4 is modified locally, Table 10 is modified on the cloud), changes are merged automatically.
     - If both systems modified the same dataset, the system applies **Temporal Last-Write-Wins (LWW)** using Mongoose `updatedAt` transaction audit logs.
     - For stock transactions, the changes are additive (decrements are pooled and subtracted relative to absolute current levels).

## 3. Printer routing & ESC/POS Setup

We support thermal print commands utilizing Raw IP Print servers (TCP port `9100` / raw sockets).

### 3.1 Network Map Configuration
Ensure thermal printers areassigned a static IP inside the local area router pool (e.g. `192.168.1.185` or similar) to prevent dynamic DHCP lease dropouts.

### 3.2 ESC/POS Formatting Byte Sequences
The system formats receipt streams dynamically. Some of the raw layout variables compiled:

| Command Action | ESC/POS Command Byte Array | Purpose |
| :--- | :--- | :--- |
| **Initialize Printer** | `ESC @` (`1B 40`) | Clears page parameters and prints clean |
| **Double Strike Text (Bold)** | `ESC G 1` (`1B 47 01`) | Emphasizes item names |
| **Double Height & Width**| `GS ! 17` (`1D 21 11`) | Title and Table Number prominence |
| **Cut Paper** | `GS V 1` (`1D 56 01`) | Fully separates target ticket |

Example sequence block generated by our internal formatting parser for a KOT print stream:
```text
\x1B\x40                // Initialize printer
\x1D\x21\x11            // Set double size height & width
  TABLE 12 - KOT
\x1D\x21\x00            // Set font to standard
------------------------
Qty   Item Name
------------------------
2     Chicken Pepperoni Pizza [Large]
1     Lemon Iced Tea
------------------------
Time: 10:45 AM
\x1D\x56\x01            // Clean guillotine paper cut
```

## 4. Hardware Setup Guide

Use this checklist during onsite franchise onboarding:

1. **Local Server Node**:
   - Deploy matching high specification terminals (Intel i3+/8GB RAM) with Ubuntu Server or Windows 11 running Docker.
   - Bind the Local IP Address statically to `192.168.1.100`.
   
2. **KDS & Captain App Handheld Access Paths**:
   - Configure tablets to connect to the internal wireless network (WiFi SSIDs should be isolated from customer guest WiFis to prevent congestion).
   - In Settings, configure the backend Endpoint URL:
     - Normal Cloud: `https://api.yourbrand.com/api/captain`
     - Local Failover: `http://192.168.1.100:3000/api/captain`

3. **Backup snap restoration**:
   - Backups are generated daily as standard Base64 zipped blocks.
   - Restores can be initiated directly via the admin portal in `Settings > Restore Snapshot`.
